# ArtCOA — Certificate of Authenticity

Статичний сайт для створення **двосторінкових сертифікатів автентичності (CoA)** з QR-перевіркою.
Працює повністю в браузері, **без сервера й бази даних**. Хостинг — GitHub Pages (безкоштовно).
Розроблено для **Smalta Art Gallery**.

**Живі адреси:**
- 🌐 Генератор: https://wizartclub-cyber.github.io/artCOA/
- ✅ Перевірка (через QR): https://wizartclub-cyber.github.io/artCOA/verify.html
- 📖 Довідка: https://wizartclub-cyber.github.io/artCOA/help.html
- 📊 Панель статистики: https://wizartclub-cyber.github.io/artCOA/admin.html

## Що робить програма

- **Сторінка 1 — Certificate of Authenticity:** усі дані твору, категорія, статус унікальності, ідентифікатори, QR-код, лінії для підписів.
- **Сторінка 2 — Artwork Identification:** зображення твору, ключові ідентифікатори та підписи.
- **PDF** генерується друком браузера → **чіткий векторний текст** (виділяється), рівно 2 сторінки A4.
- **QR-код** веде на `verify.html`; дані сертифіката «зашиті» в посилання.

### Мови
English / Українська / Slovenčina.
- **Форма** показується обраною мовою.
- На **сертифікаті** підписи: англійською для EN; двомовні (`обрана / English`) для UK і SK.

### Категорія мистецтва (Category)
Кожен твір відноситься до однієї з 8 міжнародних категорій артринку. На сертифікаті
друкується обрана категорія у форматі `КОД — Назва` (напр. `MIX — Mixed Media & Phygital Art`).

| Код | Категорія |
|---|---|
| PTG | Paintings / Живопис |
| DRW | Drawings & Works on Paper / Графіка |
| MIX | Mixed Media & Phygital Art / Змішана техніка та Фіджитал-арт |
| PRN | Prints & Multiples / Тиражна графіка |
| SCL | Sculptures & 3D Objects / Скульптура |
| PHO | Photography / Фотографія |
| DIG | Digital Art / Цифрове мистецтво |
| DEC | Design & Decorative Art / Дизайн та Декор |

При виборі категорії поле «Матеріали та техніка» показує приклад-підказку
(напр. для MIX: *Giclée on canvas, hand-embellished with acrylic and gold leaf*).

### Статус унікальності (Edition Status)
На сертифікаті явним рядком: `Unique piece (1/1)` або `Limited Edition — copy 15 of 50`.
Це головний ціноутворюючий фактор для колекціонера.

### Edition ID (формується автоматично)
`ініціали-рік-№твору-код-розмірСм-тираж` → напр. `sd-2026-001-EDH-50x70-15`.
**Номер сертифіката** = Edition ID + номер копії → `…-15-1`.

Код видання підставляється за статусом:

| Status edition | Код |
|---|---|
| Limited Edition | ED |
| Hand-embellished Limited Edition | EDH |
| Open Edition | OED |
| Original | OR |
| Artist Proof | AP |
| Digital | DIG |
| Reproduction | REP |

## Як користуватись

1. Обери мову (вгорі зліва).
2. Заповни дані твору. Обери **категорію** — поле Medium підкаже приклад. Edition ID і № сертифіката рахуються самі.
3. На стор. 2 додай зображення твору («Вибрати файл»).
4. Дата підставляється поточна (за часовим поясом користувача, формат `dd.mm.yyyy`).
5. Встав **приватний ключ підпису** (згенеруй одноразово в `keys.html`) — тримай у таємниці. Можна поставити «Запам'ятати на пристрої».
6. **Save as PDF** → у діалозі друку обери «Зберегти як PDF».
   *Порада (одноразово): More settings → Margins: None, вимкни Headers and footers.*
   **Send PDF** — поділитися готовим файлом. **Copy verify link** — скопіювати посилання.
   **Видати весь тираж** — згенерувати всі копії (1/N…N/N) одразу, кожна окремо підписана зі своїм QR, одним багатосторінковим PDF.
7. Покупець сканує QR → сторінка перевірки **автоматично** підтверджує підпис публічним ключем галереї → «Автентичний» (вводити ключ не потрібно).

## Аналітика використання

Сайт статичний, тож сам по собі не веде спільну таблицю. Збір улаштовано так:

- `analytics.js` у браузері збирає **знеособлену подію** (категорія, мова, тип девайсу),
  зберігає локально (`localStorage`) і надсилає на бекенд. **Лише після згоди користувача.**
- `admin.html` — панель перегляду (локальні події) + експорт CSV.
- `backend/apps-script.gs` — Google-таблиця = центральна таблиця всіх користувачів.

> **Збір імен покупців за замовчуванням ВИМКНЕНО** (рішення щодо GDPR — див. нижче).

### Підключити центральну таблицю
1. Виконай кроки всередині `backend/apps-script.gs` → отримаєш «Web app URL».
2. У `analytics.js` встав його у `ANALYTICS_ENDPOINT` (вже підключено для цього проекту).
3. Після зміни коду бекенда **перерозгорни**: Deploy → Manage deployments → ✏️ → New version.

## Реєстр виданих сертифікатів

Окрема вкладка **`Registry`** у тій самій Google-таблиці — **власний запис галереї** про кожен виданий сертифікат.

- Записується автоматично при **Save / Send / Видати весь тираж** — але **лише для підписаних** сертифікатів (тобто реальних видач галереї з приватним ключем; чужі/тестові не потрапляють).
- Поля: ID сертифіката, Edition ID, копія, тираж, категорія, статус, художник, назва, рік, техніка, основа, розміри, галерея, дата, **URL фото**, **посилання на перевірку**, колонка **Revoked**.
- Використовує той самий `ANALYTICS_ENDPOINT` (бекенд розрізняє за полем `kind:'registry'`). Не залежить від банера згоди (це бізнес-запис галереї, а не трекінг покупця).
- Вимкнути: `RECORD_REGISTRY = false` в `index.html`.
- **Відкликання:** познач будь-що (напр. дату) у колонці `Revoked` навпроти сертифіката → `verify.html` при перевірці звіряє ID з реєстром (`?revoked=1`) і покаже **«Відкликано»**, навіть якщо підпис дійсний. (Потрібен перерозгорнутий бекенд.)
- **Самоперевірка ключа:** генератор показує «✓ Signed» лише якщо ключ відповідає зареєстрованому ключу цього художника (`ARTIST_KEYS[ініціали]`); інакше — «mismatch» або «художника не зареєстровано».

> ⚠️ Після оновлення коду бекенда (для реєстру) **перерозгорни** Apps Script (New version). Вкладка `Registry` створиться сама.

## Фото творів (Cloudinary)

- При додаванні зображення воно **автоматично зменшується до друк-розміру** (≤1800px) у браузері,
  потім завантажується в **Cloudinary** (unsigned-upload), а URL вбудовується в **підписаний** сертифікат.
- На `verify.html` фото показується за цим URL (підміна неможлива — URL у ECDSA-підписі).
- **Налаштування:** створи безкоштовний акаунт Cloudinary + unsigned upload preset і впиши в `index.html`:
  `CLOUDINARY_CLOUD` (cloud name) та `CLOUDINARY_PRESET` (назва пресету). Без цього фото лишається лише в PDF.

## Приватність (GDPR)

- Повна **Політика приватності** — `privacy.html` (трьома мовами), посилання в банері згоди та футері.
- На сайті є **банер згоди**: збір даних відбувається **лише після «Accept»**; **«Decline»** → нічого не збирається.
- **За замовчуванням імена та IP НЕ збираються** (`COLLECT_PII = false` в `analytics.js`) — лише знеособлена
  статистика (категорія, статус, мова, девайс).
- Обробники: Cloudinary (фото), Google Sheets (аналітика, якщо увімкнено), ipwho.is (IP — лише якщо `COLLECT_PII`).

## Безпека та обмеження

- Сертифікати підписуються **асиметрично — ECDSA P-256** (WebCrypto). **У кожного художника своя пара ключів.** Підписують **приватним ключем**, а `verify.html` перевіряє **публічним ключем автоматично** — покупцю не потрібен секрет.
- **Реєстр ключів художників** `ARTIST_KEYS` (однаковий у `index.html` і `verify.html`): ключ = **ініціали** художника (ті, що в Edition ID). Сертифікат містить ініціали (вони в підписі — підмінити не можна), verify бере відповідний публічний ключ. Підробити неможливо: чужий ключ не в реєстрі.
- **Реєстр ключів** живе в одному файлі `artist-keys.js` (`window.ARTIST_KEYS`), який вантажать index і verify.
- **Апрув художника/галереї:** художник генерує пару в `keys.html`, приватний лишає собі, **публічний** + свій id (ініціали) дає галереї → у **`admin-keys.html`** галерея валідує й апрувить ключ → **Download `artist-keys.js`** → деплой (commit). Тільки задеплоєне довіряється; з QR ключі не беруться.
- Самоперевірка в генераторі: «✓ Signed» лише якщо ключ відповідає зареєстрованому ключу цього художника; інакше «mismatch» або «художника не зареєстровано».
- Обмеження: дані досі в QR-посиланні.
- **Доступ до адмінок** (`admin.html`, `admin-keys.html`) закрито паролем (`admin-gate.js`, пароль зберігається як SHA-256-хеш; дефолт `change-me-now` — **зміни його**: на гейті «Set / change passphrase» → встав новий хеш у `admin-gate.js` → деплой). Це клієнтський **стримувач**, не серверний захист; справжній контроль — доступ до GitHub-репо. Для повноцінного захисту — Cloudflare Access.
- Сторінка перевірки показує **фото твору** (з Cloudinary), якщо налаштовано хостинг; інакше фото лише в PDF.
